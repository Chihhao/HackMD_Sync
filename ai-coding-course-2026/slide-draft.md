## AI Generation & Visual Guidelines
**請 AI 在根據此大綱生成簡報內容時，嚴格遵守以下規範：**

* **嚴格頁數**：請務必依照大綱內容生成，總計 19 頁。
* **風格統一**：整份簡報需風格一致，採用**淺色底色**，保持極簡專業，避免過多裝飾。
* **內容過濾 (Content Filtering)**：
    * **禁止輸出排版參數**：投影片內容中不得出現任何字型名稱、間距設定、CSS 樣式或 Hex 色碼。
    * **禁止輸出內部指令**：絕對不要顯示 AI 內部的生成指令、排版建議或技術元數據。
    * **禁止簽名**：投影片中不得出現任何 AI 簽名或浮水印。
* **結構明確**：內容邏輯必須包含：背景挑戰、解決方案、數據支持、結論。
* **純粹內容**：所有輸出的文字必須是直接呈現給觀眾看的簡報內容，而非對排版的描述。

## Visual Style
* **設計風格**：現代專業風 (Modern Professional)。強調簡潔、高對比度與清晰的資訊層級。
* **配色方案**：
    * 主色：極簡白 (#FFFFFF) 或 淺灰 (#F8F9FA) 作為底色。
    * 強調色：企業藍 (#0056B3) 或 科技青 (#17A2B8) 用於重點標註與圖表。
    * 輔助色：深灰色 (#495057) 用於內文，中灰色 (#6C757D) 用於次要文字。
* **字體規範**：
    * 標題：無襯線體 (Sans-serif)，如 `Inter` 或 `Microsoft JhengHei UI` (粗體)。
    * 內文：等寬字體 (Monospace) 用於程式碼，如 `JetBrains Mono`；一般文字使用 `Segoe UI` 或 `PingFang TC`。
* **視覺元素**：
    * 背景浮水印：淺灰色的點狀網格 (Dot Grid) 或 簡約的幾何線條。
    * 程式碼塊：淺灰色背景 (#F1F3F5)，搭配語法高亮，模擬 GitHub 風格。

## Page 11. Context 污染與「歪樓」效應 
* 標題：Context 污染：AI 為什麼會「歪樓」？
* 副標題：AI 的「即時記憶」與「理解邊界」
* 視覺設計：背景維持極簡白 (#FFFFFF) 搭配淺灰點狀網格。採用「動態流程示意圖」。中央繪製一個垂直排列的對話串列（Context Window）。其中一塊 AI 回覆顯示為紅色警示色（代表幻覺），其下方的後續對話方塊隨之變形、歪斜（象徵 Context 污染）。右側演示「修正流程」：一個現代風格的垃圾桶圖示移除紅色方塊後，後續對話流即恢復整齊排列與「科技青 (#17A2B8)」的正常色調。
* 主要內容：
    * Garbage In, Garbage Out：AI 的記憶是「盲目」的，它會把之前的錯誤當成真理。
    * 預防「歪樓」：一旦 AI 開始胡說八道，該錯誤就會污染整個 Context。
    * 主動管理：善用「刪除單則訊息」與「編輯」，保持 Context 的純淨。
* 講稿：大家要記住一個核心觀念：Garbage In, Garbage Out。AI 其實沒有判斷是非的能力，它只會根據窗口裡的內容來推測下一句話。如果 AI 剛才給了一個錯誤的 API 或邏輯，而你沒有刪掉它就繼續問，AI 可能就會以為那個錯誤是正確的背景，導致整串對話開始「歪樓」。所以，發現 AI 答非所問時，**不要試圖糾正它**，最有效的方法是 **直接刪掉** 那則回覆，讓 Context 回到乾淨的狀態。

## Page 12. Token 量化與 600KB 的秘密 
* 標題：Token：AI 的計量單位
* 副標題：為什麼小旺限制 600KB？
* 視覺設計：背景維持極簡白 (#FFFFFF) 搭配淺灰點狀網格。中央展示一個現代化的「容量條 (Capacity Bar)」或「水平堆疊長條圖」。左段以「科技青 (#17A2B8)」實心區塊代表「600KB 程式碼 (約 15-20 萬 Tokens)」，右段以淺灰虛線區塊代表「剩餘空間 (對話歷史 + 系統提示詞)」。上方清楚標示 GPT-5 總容量 (27 萬 Tokens)，並以圖示強調若程式碼佔比過大將擠壓「短期記憶」空間。
* 主要內容：
    * **換算公式**：1000 Tokens ≈ 750 英文字；600KB 程式碼 ≈ 15~20 萬 Tokens。
    * **精準選檔**：600KB 是為了確保 AI 在理解代碼的同時，還有足夠的「短期記憶」來跟你對話。
    * **避免溢位**：若代碼佔滿 27 萬 Token，AI 會立刻忘記你五分鐘前說過的指令。
* 講稿：大家常問為什麼小旺限制只能選 600KB 的檔案？我們來算一下：600KB 的純文字代碼大約會消耗 15 到 20 萬個 Token。雖然我們用的 GPT-5 有 27 萬的容量，但我們必須留下一部分空間給「系統提示詞」和「後續的對話紀錄」。如果代碼塞太滿，AI 很快就會出現「失憶」現象，忘記你剛才要求的修改規則。所以，精準挑選相關檔案，才是最高效的做法。

## Page 17. AI 小旺：Coding 模式實戰
* 標題：如何高效使用 AI 小旺
* 副標題：從讀取到「一鍵套用」的完整閉環
* 視覺設計：背景維持極簡白 (#FFFFFF) 搭配淺灰點狀網格。採用「Context 刷新機制對比圖」。
    * **左側 (一般模式)**：以灰階呈現對話框內堆疊了多個版本的代碼塊，標註「資訊冗餘」。
    * **右側 (Coding 模式)**：以「科技青 (#17A2B8)」高亮中央固定的「代碼槽位」，並用動態圖示象徵本機檔案更新時，槽位內容直接替換為「最新代碼」，標註「自動同步」。
    * **下方**：簡潔的操作流程截圖（啟動器 -> 勾選 -> Diff 預覽 -> Apply），強調一氣呵成的體驗。
* 主要內容：
    * 挑選檔案：只選相關檔案，避免雜訊。
    * 排除巨型檔案：不要勾選 .pat 或 .tim 檔。
    * 600KB 上限：精準控制 Context 大小。
    * **自動同步最新代碼**：每一回合送出時重新讀取檔案，確保 AI 永遠基於最新版本推理。
    * **一鍵套用 (Apply Patch)**：直接在網頁預覽變更並寫回本機檔案。
    * **硬體安全審查**：針對 `WriteRegister` 等關鍵字，Apply 前必須人工確認時序。
* 講稿：Coding 模式是小旺的核心。它有一個隱藏的大優點：**自動更新記憶**。傳統對話如果你改了 Code 再貼一次，AI 的 Context 裡會同時存在新舊兩個版本，很容易讓 AI 產生幻覺。但在 Coding 模式下，每一回合送出時，小旺都會重新抓取硬碟裡最新的檔案內容，舊的代碼會被自動替換掉。這確保了 AI 永遠是在最新的基礎上幫你往下改。再加上「一鍵套用」功能，確認 Diff 沒問題後點下 Apply 即可。但要提醒大家，**邏輯正確不代表硬體行為正確**，看到涉及暫存器寫入的修改，請務必對照 Manual 再按下去。
