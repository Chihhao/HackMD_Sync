## Page 4. 資安第一：脫敏與 IP 保護
* 標題：資安意識：如何安全地餵資料？

* 講稿：大家要有一個正確的認知：雖然「AI 小旺」是內部開發的工具，但它背後串接的其實是 OPENAI 的 API。這意味著你傳過去的資料會離開公司。所以，資安意識是工程師的底線。大家一定要養成「脫敏」的習慣：把專案代號改成 Project_A，把具體的良率數字改成變數。特別是涉及晶片核心架構的文件，千萬不要整份丟進去，我們只需要擷取那段讓你頭痛的程式邏輯就好。

## Page 5. Markdown：AI 的母語
* 標題：基礎工具：Markdown 與 HedgeDoc
* 副標題：溝通與紀錄的標準格式
* 視覺設計：對比與流程圖。
    * 左側：Word/Excel/PPT 圖示被打叉，Markdown 圖示被勾選（標註：AI Friendly）。
    * 右側：小旺匯出按鈕 -> 箭頭 -> HedgeDoc 介面（標註：一鍵生成報告）。
* 主要內容：

    * **數位轉型**：習慣用 Markdown 記錄大小事，取代傳統 Word/PPT，為 AI 協作打下基礎。
* 講稿：在開始寫 Prompt 之前，我們要先學會 AI 的母語——Markdown。為什麼要學？因為 Markdown 是純文字，AI 讀起來最輕鬆，不像 Word 或 PPT 有一堆複雜的隱藏格式。我建議大家以後記錄大小事都改用 Markdown，這樣以後要餵給 AI 處理時會非常方便。另外，小旺支援「整串匯出」功能，你可以直接把對話內容轉貼到 HedgeDoc，一秒鐘就能變出漂亮的技術報告，這才是高效工程師的作法。

## Page 6. Markdown 基礎語法速成
* 標題：Markdown 基礎語法
* 副標題：工程師必備的標記語言
* 視覺設計：左右對照圖。左側是原始碼（Raw），右側是渲染後的結果（Rendered）。
* 主要內容：

    * **文字樣式**：`**粗體**`、`*斜體*`、` `行內程式碼` `。

    * **圖表預告**：簡單提到 Mermaid 語法可以畫流程圖。
* 講稿：Markdown 語法非常直覺。對我們來說，最重要的是「程式碼區塊」。當你把 Code 餵給 AI，或是 AI 回覆你時，都會用三個反引號包起來。學會這個，你就能在 HedgeDoc 裡排版出非常漂亮的技術文件，甚至還能用 Mermaid 語法叫 AI 幫你畫流程圖。

## Page 10. 語言掌握度差異：Assembly vs. ATL
* 標題：AI 的語言天賦：為什麼它會張冠李戴？
* 副標題：理解訓練資料對 AI 能力的影響

## Page 11. 深入理解 Context (上下文) 
* 標題：什麼是 Context (上下文)？
* 副標題：AI 的「即時記憶」與「理解邊界」
* 視覺設計：延續「滑動窗口」示意圖，但加入「污染」元素。
    * 窗口中有一則紅色的 AI 回覆（標註為「錯誤資訊/幻覺」）。
    * 後續產出的 AI 方塊也隨之變紅且歪斜，代表「歪樓」。
    * 旁邊出現一個巨大的「垃圾桶」圖示，將紅色方塊移除後，後續回覆恢復正常。
* 主要內容：
    * Garbage In, Garbage Out：AI 的記憶是「盲目」的，它會把之前的錯誤當成真理。
    * 預防「歪樓」：一旦 AI 開始胡說八道，該錯誤就會污染整個 Context。
    * 主動管理：善用「刪除單則訊息」與「編輯」，保持 Context 的純淨。
* 講稿：大家要記住一個核心觀念：Garbage In, Garbage Out。AI 其實沒有判斷是非的能力，它只會根據窗口裡的內容來推測下一句話。如果 AI 剛才給了一個錯誤的 API 或邏輯，而你沒有刪掉它就繼續問，AI 可能就會以為那個錯誤是正確的背景，導致整串對話開始「歪樓」。所以，發現 AI 答非所問時，**不要試圖糾正它**，最有效的方法是 **直接刪掉** 那則回覆，讓 Context 回到乾淨的狀態。

## Page 12. Token 量化與 600KB 的秘密 
* 標題：Token：AI 的計量單位
* 副標題：為什麼小旺限制 600KB？
* 視覺設計：對比圖表。
    * 換算公式：1000 Tokens ≈ 750 個英文字；**600KB 程式碼 ≈ 15~20 萬 Tokens (約 1.2~1.5 萬行)**。
    * 剩餘空間：展示 GPT-5 的 27 萬總容量，扣除 600KB 代碼後，剩下的空間留給「對話歷史」與「系統提示詞」。
* 主要內容：
    * **精準選檔**：600KB 是為了確保 AI 在理解代碼的同時，還有足夠的「短期記憶」來跟你對話。
    * **避免溢位**：若代碼佔滿 27 萬 Token，AI 會立刻忘記你五分鐘前說過的指令。
* 講稿：大家常問為什麼小旺限制只能選 600KB 的檔案？我們來算一下：600KB 的純文字代碼大約會消耗 15 到 20 萬個 Token。雖然我們用的 GPT-5 有 27 萬的容量，但我們必須留下一部分空間給「系統提示詞」和「後續的對話紀錄」。如果代碼塞太滿，AI 很快就會出現「失憶」現象，忘記你剛才要求的修改規則。所以，精準挑選相關檔案，才是最高效的做法。

## Page 16. AI 對話的進階技巧
* 標題：與 AI 溝通的進階武器
* 副標題：讓 AI 輸出更精準、更有邏輯

## Page 17. AI 小旺：Coding 模式實戰
* 標題：如何高效使用 AI 小旺
* 副標題：從讀取到「一鍵套用」的完整閉環
* 視覺設計：**「Context 刷新機制」對比圖**。
    * **左側 (一般模式)**：對話框內堆疊了「代碼 V1」、「代碼 V2」、「代碼 V3」，標註「資訊冗餘、容易混淆」。
    * **右側 (Coding 模式)**：中央有一個固定的「代碼槽位」，當本機檔案更新時，槽位內的內容會閃爍並直接替換為「最新代碼」，標註「自動同步、邏輯穩定」。
    * **下方**：操作步驟截圖（啟動器 -> 勾選檔案 -> Diff 預覽 -> Apply）。
* 主要內容：
    * 挑選檔案：只選相關檔案，避免雜訊。
    * 排除巨型檔案：不要勾選 .pat 或 .tim 檔。
    * 600KB 上限：精準控制 Context 大小。
    * **自動同步最新代碼**：每一回合送出時重新讀取檔案，確保 AI 永遠基於最新版本推理。
    *   **一鍵套用 (Apply Patch)**：直接在網頁預覽變更並寫回本機檔案。
    *   **硬體安全審查**：針對 `WriteRegister` 等關鍵字，Apply 前必須人工確認時序。
* 講稿：Coding 模式是小旺的核心。它有一個隱藏的大優點：**自動更新記憶**。傳統對話如果你改了 Code 再貼一次，AI 的 Context 裡會同時存在新舊兩個版本，很容易讓 AI 產生幻覺。但在 Coding 模式下，每一回合送出時，小旺都會重新抓取硬碟裡最新的檔案內容，舊的代碼會被自動替換掉。這確保了 AI 永遠是在最新的基礎上幫你往下改。再加上「一鍵套用」功能，確認 Diff 沒問題後點下 Apply 即可。但要提醒大家，**邏輯正確不代表硬體行為正確**，看到涉及暫存器寫入的修改，請務必對照 Manual 再按下去。

* 標題：從 Coder 轉型為 Reviewer

* 視覺設計：簡潔的專業工作空間。工程師正在審查螢幕上的程式碼與測試數據，背景隱約可見 ATE 測試機台，強調「專業審核」與「硬體環境」的結合。
* 主要內容：
    * 角色轉變：你是架構師與審查者。
    * 核心競爭力：領域知識 (Domain Knowledge) 與問題定義能力。
    * **安全底線：外部 API 脫敏是不可逾越的紅線。**
    * 資源彙整：小旺連結、HedgeDoc 連結。
* 講稿：總結今天的課程，AI 是一個強大的生產力工具，但它無法取代你的專業判斷。它能幫你快速生成邏輯與樣板，但只有你了解硬體的物理限制、時序要求與測試策略。你的價值在於定義正確的問題，並對 AI 的輸出進行嚴格的審查與驗證。請記住：AI 負責執行，你負責決策。最後再次提醒：**雖然工具是內部的，但 API 是外部的，脫敏永遠是你的安全底線**。讓我們一起利用 AI 實現更高效的開發。謝謝大家！
